# Arkitekturaren ikuspegi orokorra

## Eredu arkitektonikoa

Trainium-ek aurkezpen-geruza (konposagarriak + ViewModels), datuetarako biltegi-geruza eta zerbitzu-kokatzailea injekzio-puntu gisa konbinatzen dituen geruza-arkitektura jarraitzen du. Negozio-logika gehiena ViewModels eta biltegietan dago; aurkezpen-geruzak deien bidez delegatzen ditu eta haien egoera erreaktiboa behatzen du.

```
Capa de presentación  →  ViewModels  →  Repositorios  →  SupabaseClient / DataConnectClient
```

Bigarren mailako pantaila batzuek (historia, premium) oraindik koroutinak zuzenean abiarazten dituzte `rememberCoroutineScope()`-n, baina fluxu nagusiak (makinak, erreserbak, autentifikazioa, profila) dagoeneko ViewModel + Repository ereduaren bidez funtzionatzen du.

## Kanpoko backendak

Aplikazioak kanpoko bi backendekin elkarreragiten du:

**Supabase** backend nagusi gisa jokatzen du. Datu-taula guztiak kudeatzen ditu PostgREST protokoloa erabiliz. Supabase Kotlin SDK-k automatikoki mapatzen ditu JSON erantzunak datu-klaseekin `kotlinx.serialization` erabiliz. Autentifikazioa zuzenean `usuarios` taularen aurka egiten da IDa eta pasahitza egiaztatuz, ez Supabase Auth fluxaren bidez (haren modulua barne dagoen baina aktiboki erabiltzen ez dena).

**Firebase Data Connect** backend osagarri gisa erabiltzen da GraphQL amaierako puntua deitzen duen eskuzko HTTP bezero baten bidez (`DataConnectClient.kt`). Firebase Auth SDK menpekotasun gisa deklaratu da eta erabilgarri dago autentifikazio-sistemaren migraziorako.

## Estatuaren kudeaketa

UI egoera bi modutan kudeatzen da pantailaren arabera:

**ViewModels (eredu nagusia):** Konplexutasun handiagoa duten pantailek (makinak, erreserbak, platerak, saioa hasteko, erregistroa, profila) beren egoera erakusten dute ViewModelen adierazitako eta konposagarritik behatutako `mutableStateOf` aldagaien bidez. ViewModel-ek pantailako biraketak eta konfigurazio-aldaketei eusten die.

**Lokal egoera (bigarren mailako pantailak):** Pantaila sinple batzuek (historia, premium) `remember` + `mutableStateOf` erabiltzen dute konposagarriaren barruan. Kasu hauetan, egoera ez da pantailaren biraketari eusten, eta hori onargarria da aplikazioak diseinuaren arabera orientazio erretratua behartzen baitu.

Gaiaren hobespena diskoan mantentzen da `ThemeManager` (DataStore) eta `MainActivity.kt`-n irakurtzen da `Flow<Boolean?>` `collectAsState`-rekin bildutako __IC_18__ gisa.

Hizkuntza-hobespena `LanguageManager`-k (DataStore) mantentzen du eta `Flow<AppLanguage?>` gisa ebazten du `MainActivity.kt`. Erabiltzaileak ez badu hizkuntzarik hautatu, sistemaren hizkuntza `languageManager.detectSystemLanguage()`-rekin detektatzen da.

## Nazioartekotzea (i18n)

Nazioartekotze-sistemak `CompositionLocal` (`LocalStrings`) erabiltzen du konposagarri guztiei hizkuntza aktiboaren testu-kateetarako sarbidea emateko, kateak parametro gisa pasatzeko beharrik gabe.

### Onartutako hizkuntzak

| Kodea | Hizkuntza | 
|---|---| 
| `es` | Gaztelania | 
| `en` | ingelesa | 
| `ca-VALENCIA` | Valentziarra | 
| `ca` | katalana | 
| `fr` | frantsesa | 
| `ar-MA` | Marokoko arabiera | 
| `eu` | euskara | 
| `gl` | galiziera | 
| `de` | alemana | 
| `pt` | portugesa | 
| `ru` | errusiera | 
| `zh` | txinera |

### Hizkuntza-ebazpen-fluxua

```
LanguageManager.currentLanguage (Flow<AppLanguage?>)
    ├── null → languageManager.detectSystemLanguage() (Locale del sistema)
    └── AppLanguage → stringsForLanguage(language) → AppStrings
                                    ↓
                    CompositionLocalProvider(LocalStrings provides currentStrings)
                                    ↓
                        val strings = LocalStrings.current  // en cualquier composable
```

Hizkuntza-hautatzailea `ProfileScreen`, `EditProfileScreen`, `MaquinasScreen` eta beste pantaila nagusi batzuetan eskuragarri dagoen mundu-ikonotik irekitzen da. `LanguagePickerDialog` hizkuntza bat hautatzean `LanguageManager.setLanguage(language)`-rekin kodea mantentzen da eta `MainActivity` konposagarriak kateak automatikoki birkargatzen ditu.

## Jakinarazpen sistema

Aplikazioak tokiko erreserba-abisuak programatzen ditu `AlarmManager` erabiliz. Sistemak hiru osagai ditu:

- **`NotificationHelper`:** Jakinarazpen kanala sortzen du, alarma zehatzak programatzen ditu (`setExactAndAllowWhileIdle`) eta bertan behera uzten ditu. `rescheduleNextReservationNotification` ere badu, erabiltzailearen etorkizuneko erreserbak kontsultatzen dituena eta hurrengorako alarma berriro programatzen duena. 
- **`NotificationReceiver`:** `BroadcastReceiver` alarma jasotzen duena eta jakinarazpena `NotificationCompat`-rekin argitaratzen duena. 
- **`NotificationSettingsManager`:** Erabiltzaileak aukeratutako aldez aurretiko oharrak (aukerak: 10, 15, 30, 60 edo 120 minutu; balio lehenetsia: 30 minutu) eta jakinarazpenen aktibazio-egoera globalak (lehenespenez aktibatuta) DataStore-n jarraitzen dute.

`POST_NOTIFICATIONS` baimena Android 13+en exekuzioan eskatzen da `MainActivity.onCreate()` geroztik. `SCHEDULE_EXACT_ALARM` eta `USE_EXACT_ALARM` baimenak manifestuan adierazten dira, alarmen zuzentasuna ziurtatzeko.

## Mendekotasun injekzioa

`ServiceLocator` zerbitzu-lokatzaileak (`di/ServiceLocator.kt`-n) biltegiak `lazy` inprimakian instantziatzen ditu (biltegi bakoitzeko instantzia bakarra, lehen sarbidean sortua):

```kotlin
object ServiceLocator {
    val authRepository by lazy { AuthRepository() }
    val usuarioRepository by lazy { UsuarioRepository() }
    val maquinaRepository by lazy { MaquinaRepository() }
    val reservaRepository by lazy { ReservaRepository() }
    val platoRepository by lazy { PlatoRepository() }
    val pesoRepository by lazy { PesoRepository() }
    val pagoRepository by lazy { PagoRepository() }
}
```

ViewModels-ek biltegietara sartzen dira `ServiceLocator` bidez, geruzen arteko akoplamendu solteak mantenduz.

## Konfigurazio zentralizatua

`AppConfig.kt` aplikazioen konfigurazio-konstante guztiak zentralizatzen ditu:

```kotlin
object AppConfig {
    const val SUPABASE_URL = "https://<tu-proyecto>.supabase.co"
    const val SUPABASE_KEY = "<tu-anon-key>"
    const val FIREBASE_API_KEY = "<tu-api-key>"
    const val FIREBASE_APP_ID = "<tu-app-id>"
    const val DATACONNECT_URL = "https://europe-southwest1-dataconnect.googleapis.com/..."
    const val SESSION_PREFS_NAME = "trainium_session"
    const val SESSION_DURATION_MS = 7L * 24L * 60L * 60L * 1000L  // 7 días
    const val IMAGE_QUALITY = 70
    // Formatos de fecha...
}
```

Era berean, `DbConstants.kt` `DbTables` eta `DbColumns` objektuak taulen eta zutabeen izenekin definitzen ditu konstante gisa, kodean zehar sakabanatuta dauden kate literalak saihestuz.
